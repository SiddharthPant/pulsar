# NATS-backed session authentication plan

## Goal

Add simple email/password authentication to the Axum application, with:

- PostgreSQL as the source of truth for users and password hashes.
- `axum-login` and `tower-sessions` for authentication and HTTP session lifecycle.
- NATS JetStream Key/Value as the persistent session store.
- Core NATS subjects for live SSE fan-out.
- Separate JetStream streams and durable consumers for background jobs.
- Ordinary server-rendered HTML forms first, with Datastar used only for optional progressive enhancement.

The browser will hold only an opaque session ID in a secure cookie. It must never receive password hashes, serialized user data, or a bearer token accessible to JavaScript.

## Important architectural constraint

Use JetStream Key/Value for sessions, not Core NATS subjects. Core NATS is transient and unsuitable for persistent session state.

NATS KV does not guarantee read-your-writes when a clustered bucket serves direct reads from followers or mirrors. This can create a short stale-read window immediately after login, logout, password change, or session revocation.

- The initial single-node deployment is acceptable.
- Before enabling a multi-node production NATS cluster, prove that session reads are routed to the stream leader or otherwise provide the consistency required for immediate revocation.
- If that cannot be guaranteed, keep the `SessionStore` boundary and replace only the NATS implementation with PostgreSQL or Redis.
- Authentication must fail closed when the session store is unavailable.

## Target NATS topology

```text
Core NATS
└── Live SSE fan-out
    └── events.user.<user-id>

JetStream streams
└── Background jobs
    ├── jobs.email
    └── jobs.cleanup

JetStream KV
└── PULSAR_SESSIONS
    └── <opaque-session-id> -> serialized tower_sessions::Record
```

Sessions, SSE events, and background jobs must use separate subjects, streams/buckets, retention policies, and permissions.

## Phase 1: introduce NATS infrastructure

### Docker Compose

Update `compose.yml` with a NATS service that:

- Uses an explicitly pinned NATS image.
- Enables JetStream with `-js`.
- Stores JetStream data under `/data` using `-sd /data`.
- Persists `/data` in a named volume.
- Exposes port `4222` for clients.
- Optionally exposes the monitoring endpoint on `8222` for local development.
- Has a health check suitable for the chosen image.

Development settings may run without credentials on loopback. Production must use TLS and NATS credentials/NKeys.

### Application configuration

Extend `apps/server/src/config.rs` with:

- `NATS_URL`, defaulting locally to `nats://127.0.0.1:4222`.
- `NATS_SESSION_BUCKET`, defaulting to `PULSAR_SESSIONS`.
- An environment-dependent secure-cookie flag, or an equivalent explicit application-environment setting.
- Session inactivity and maximum-lifetime settings if they need to be configurable.

Update `mise.toml` with NATS lifecycle tasks and make local development start/wait for both PostgreSQL and NATS before launching the server.

### Dependencies

Add compatible, explicitly pinned workspace dependencies for:

- `async-nats`, with `jetstream` and `kv` features.
- `async-trait` if required by the selected `tower-sessions` version.
- `axum-login`.
- `argon2`.
- `time`.

Import the `tower-sessions` types re-exported by `axum-login` to ensure the custom store implements the exact version of the `SessionStore` trait used by the authentication layer.

Keep SQLx 0.9. The NATS session store removes the need for the SQLx-based Tower session-store crate.

## Phase 2: implement the NATS session store

Create `apps/server/src/session_store.rs` containing a small adapter:

```rust
#[derive(Clone, Debug)]
pub struct NatsSessionStore {
    bucket: async_nats::jetstream::kv::Store,
}
```

Implement the Tower `SessionStore` operations as follows:

| Tower operation | NATS KV operation | Required behavior |
| --- | --- | --- |
| `create` | `create` | Atomic create; regenerate and retry on an ID collision. |
| `save` | `put` | Serialize and replace the current record. |
| `load` | `get` | Deserialize; reject and purge an expired record. |
| `delete` | `purge` | Remove the value and its historical revisions on logout/revocation. |

Requirements:

- Override `create`; do not implement new-session creation with `put`.
- Use an opaque, cryptographically random session ID generated by Tower Sessions.
- Use the session ID string as the KV key.
- Serialize the complete `tower_sessions::Record` with a stable format such as JSON or MessagePack.
- Map NATS and serialization failures to session-store backend errors without leaking details to HTTP clients.
- Check `Record.expiry_date` on every load. An expired record is invalid even if JetStream has not physically removed it.
- Treat a missing key as no session.
- Treat a NATS timeout, permission error, or connection failure as a backend failure, not as an anonymous user.

Tower Sessions warns that concurrent read-modify-write operations may lose updates. Keep authentication session state small and avoid counters or unrelated mutable application state in the session. If later session mutations require concurrency control, add revision-aware compare-and-set semantics.

### KV bucket configuration

Create or update the bucket during application startup with approximately:

- Name: `PULSAR_SESSIONS`.
- Storage: file.
- History: `1`.
- Maximum age: the longest allowed absolute session lifetime, initially 30 days.
- Maximum value size: approximately 8 KiB.
- Maximum bucket size: a deliberate operational limit, initially approximately 64 MiB.
- Replicas: `1` in the local single-node environment; `3` only after the clustered consistency issue is resolved.

Bucket `max_age` is garbage collection, not the source of truth for validity. The record expiry remains authoritative. Sliding expiry rewrites the record, refreshing its bucket age.

Do not depend on NATS per-message TTL in the first implementation. Application-level expiry is simpler and works consistently through the KV API.

### Startup and shutdown

In `apps/server/src/main.rs`:

1. Connect to PostgreSQL.
2. Connect to NATS.
3. Create the JetStream context.
4. Create or update the session KV bucket.
5. Construct `NatsSessionStore`.
6. Construct the Tower session and Axum Login layers.
7. Start serving only after all required dependencies are ready.

If PostgreSQL or NATS initialization fails, startup should fail with contextual logs. Reuse a single cloned `async_nats::Client`; it is intended to be shared.

## Phase 3: add user authentication

### Database migration

Add a new reversible migration rather than editing `20260716150902_create_users_table.sql`.

The migration should:

- Add `password_hash text` to `users`.
- Add a case-insensitive unique constraint/index for normalized email addresses.
- Account for existing users that do not yet have passwords.

Use a staged rollout if data already exists:

1. Add `password_hash` as nullable.
2. Backfill/reset existing accounts.
3. Make the field `NOT NULL` in a later migration.

Normalize email consistently before both insert and lookup. Confirm that existing data contains no case-insensitive duplicates before creating the unique index.

### Authentication module

Use `apps/server/src/auth.rs` for:

- The authenticated user model.
- Login credentials.
- A PostgreSQL-backed `AuthnBackend`.
- The `AuthUser` implementation.
- Password hashing and verification helpers.

Use the password hash as `session_auth_hash` so changing a password invalidates existing sessions.

Authentication behavior:

- Hash passwords with Argon2id and a unique generated salt.
- Perform hashing and verification in `tokio::task::spawn_blocking`.
- Tune Argon2 parameters on deployment hardware while meeting current OWASP minimums.
- Normalize the submitted email.
- Return the same public response for an unknown email and a wrong password.
- Perform a dummy Argon2 verification for an unknown account to reduce timing-based enumeration.
- Never log credentials, password hashes, session IDs, or CSRF tokens.

### Handlers and templates

Create:

- `GET /login` to render the login page.
- `POST /login` to authenticate, cycle/create the session ID, and redirect.
- `POST /logout` to revoke the server-side session, clear the cookie, and redirect.
- `apps/server/templates/login.html`.

Start with an ordinary HTML form using `application/x-www-form-urlencoded`. This preserves browser navigation, redirects, password-manager behavior, accessibility, and operation without JavaScript.

If a `next` parameter is supported, accept only a validated same-origin absolute path beginning with `/`; reject schemes, hosts, and protocol-relative values.

### Routing

In `apps/server/src/router.rs`, split routes into public and protected groups:

- Public: `/`, `/login`, and `/assets`.
- Protected: `/users` and future authenticated application routes.
- Apply `login_required!` only to protected routes.
- Apply the authentication/session layer around both public and protected routes so login/logout handlers can access it.

Full browser navigations may redirect unauthenticated users to `/login`. Datastar requests should eventually receive an explicit unauthorized response/navigation instruction rather than accidentally patching the login document into an unrelated part of the current page.

### Cookie policy

Production cookie:

- Name: `__Host-session`.
- `Secure=true`.
- `HttpOnly=true`.
- `SameSite=Lax` or `SameSite=Strict`.
- `Path=/`.
- No `Domain` attribute.

Local plain-HTTP development must use a different non-`__Host-` cookie name with `Secure=false`, because browsers enforce the `__Host-` requirements.

Initial expiry policy:

- No “remember me”.
- Inactivity expiry: choose 24 hours or 7 days and document the choice.
- Absolute maximum: 30 days.
- Renew the session ID after authentication and any future privilege change.

### CSRF

Cookie authentication requires CSRF protection for unsafe methods.

- Make logout a `POST`, never a `GET`.
- Add a synchronizer CSRF token or equivalent well-reviewed protection for authenticated state-changing requests.
- Validate request origin as an additional defense.
- Keep `SameSite` enabled, but do not treat it as the only CSRF defense.
- If Datastar later submits a protected action, send the CSRF token as a form field or explicit request header.

## Phase 4: integrate Datastar safely

Datastar is not responsible for authentication. The browser automatically includes the same-origin `HttpOnly` session cookie; Datastar must not read it.

Datastar may later enhance:

- Inline login errors.
- Pending/disabled submit state.
- Logout navigation.
- CSRF-header submission.

Rules:

- Do not store passwords, session IDs, bearer tokens, or CSRF tokens in persisted Datastar signals.
- Prefer a normal HTML login form initially.
- If enhancing the form, use Datastar form encoding rather than its default behavior of sending all ordinary signals as JSON.
- Do not use `localStorage`, `sessionStorage`, or `data-persist` for credentials.

## Phase 5: SSE and background jobs

Implement these after the NATS connection and auth boundary are stable.

### SSE

- Use Core NATS for ephemeral live fan-out when missed events do not need replay.
- Namespace subjects by authenticated user, tenant, or resource.
- Authorize the requested subscription before creating it.
- Never accept a client-supplied subject without mapping it through server authorization rules.
- Validate the session when establishing the SSE response.
- Bound the connection lifetime by session expiry.
- Decide how logout or account revocation terminates existing long-lived SSE connections; deleting the session does not automatically close them.

Use JetStream instead of Core NATS only where reconnect/replay semantics are required.

### Background jobs

- Use a dedicated JetStream stream and durable pull consumers.
- Acknowledge only after successful processing.
- Use negative acknowledgements or delayed retries for transient failures.
- Set a maximum delivery count and define a dead-letter/advisory policy.
- Make every handler idempotent because JetStream delivery is at least once.
- Keep job subjects and permissions separate from session KV subjects.

## Testing plan

### Unit tests for `NatsSessionStore`

- Atomic creation succeeds.
- An existing key causes collision handling rather than overwrite.
- Save followed by load returns the same record.
- Missing key returns `None`.
- Expired record returns `None` and is purged.
- Delete/purge invalidates the record.
- Corrupt serialized data produces a backend error.
- NATS unavailability produces a backend error.

### HTTP integration tests

- Anonymous `/users` request redirects to `/login`.
- Valid credentials create a session cookie and grant access.
- Invalid email and invalid password return indistinguishable public responses.
- The cookie has the intended production attributes when secure mode is enabled.
- Login rotates the pre-authentication session ID.
- Logout clears the cookie and purges the NATS record.
- A logged-out session cannot access protected routes.
- Password change invalidates existing sessions through `session_auth_hash`.
- Expired sessions are rejected.
- Unsafe requests without a valid CSRF token are rejected.
- A NATS outage does not bypass authentication and returns a controlled failure.

### NATS integration tests

Run tests against an isolated NATS server with JetStream enabled. Use a uniquely named bucket per test process or suite, and delete it during teardown. Avoid sharing the developer session bucket.

Before clustered production deployment, add an explicit test for:

1. Create session.
2. Immediately read from another application instance.
3. Delete/revoke session.
4. Immediately read from another application instance.
5. Verify the required consistency and revocation behavior under node failure and leader changes.

## Observability and operations

Add metrics/logs for:

- NATS connection state and reconnects.
- Session-store operation latency and failures by operation.
- Authentication success/failure counts without user-identifying labels.
- Active SSE connections.
- Job backlog, retry count, maximum-delivery exhaustion, and processing latency.

Do not log raw KV keys because they are session IDs.

Operational requirements:

- Back up JetStream storage only if business requirements demand session restoration; losing sessions should log users out, not lose primary business data.
- Set explicit JetStream memory/file limits.
- Use local SSD storage for production JetStream nodes, not a shared NFS/NAS directory.
- Use three JetStream nodes for high availability; one volume belongs to exactly one node.
- Document that NATS is now on the critical request path for authenticated traffic.

## Suggested implementation order

1. Add NATS to Compose, configuration, and development tasks.
2. Add the Async NATS client and create the session KV bucket at startup.
3. Implement and unit-test `NatsSessionStore`.
4. Wire `tower-sessions` and `axum-login` using the custom store.
5. Add the password-hash migration and authentication backend.
6. Add login/logout handlers and templates.
7. Split public and protected routes.
8. Add cookie hardening and CSRF protection.
9. Add HTTP integration tests, including NATS failure behavior.
10. Add Core NATS SSE fan-out.
11. Add JetStream background-job streams and workers.
12. Resolve and test clustered KV consistency before enabling replicated production sessions.

## Definition of done

- Users authenticate using an Argon2id password hash stored in PostgreSQL.
- The browser stores only an opaque hardened session cookie.
- Sessions persist in the `PULSAR_SESSIONS` JetStream KV bucket.
- New sessions use atomic create and cannot overwrite a colliding session ID.
- Expired and revoked sessions cannot access protected routes.
- Authentication fails closed when NATS cannot be queried.
- Logout purges the server-side session and clears the cookie.
- Password changes invalidate old sessions.
- Unsafe authenticated requests have CSRF protection.
- The full automated test suite covers the session store and HTTP auth lifecycle.
- Session data and subject permissions are isolated from SSE and background-job infrastructure.
- Production deployment documentation explains the NATS availability and consistency requirements.
